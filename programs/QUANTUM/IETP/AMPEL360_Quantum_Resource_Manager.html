0   Introduction

0.1   Purpose

This Interactive Electronic Technical Publication (IETP) defines the complete specification for the AMPEL360 Quantum Resource Manager (QRM) — the subsystem responsible for managing Quantum Processing Unit (QPU) resources, circuit execution scheduling, telemetry monitoring, and quantum-classical hybrid orchestration across all AMPEL360 platform variants (Q100, Q250, PLUS, PLUS+, AC-MACH).

0.2   Scope

The QRM interfaces with the universal quantum platform binary (AMPEL360-UNIVERSAL-002_Quantum_Platform.bin) and operates under the Core OS real-time kernel. The reference implementation resides at src/quantum/resource_manager.py. This document is structured into four subsystem specifications, each corresponding to a major functional area depicted in Figure 1 below.

1   QPU Allocation and Scheduling Subsystem

1.1   Functional Overview

The QPU Allocation subsystem is responsible for selecting the optimal Quantum Processing Unit for each submitted circuit job based on a composite scoring algorithm. It maintains a live registry of all available QPUs, their operational status, and current workload. The scheduling flow is depicted in Figure 2.

As shown in Figure 2, when no QPU is available the job is re-queued and will be retried during the next scheduling pass initiated by schedule_pending().

1.2   Scoring Algorithm

The allocator computes a composite score S(q, j) for each candidate QPU q and circuit job j:

S(q, j) = w_p · U_lvl(j)  +  w_f · F_avg(q)  -  w_e · E_rate(q)  -  w_d · C_depth(j)
U_lvl(j)
— Normalised user priority level for job j. Mapped from the CircuitPriority enum (0 = CRITICAL … 4 = BACKGROUND) to a continuous range [1.0 … 0.0] via 1.0 − (priority / 4.0).
F_avg(q)
— Average gate fidelity of QPU q, ranging 0.0 – 1.0. Sourced from the latest calibration telemetry.
E_rate(q)
— Current aggregate error rate of QPU q, ranging 0.0 – 1.0. Higher values penalise the score.
C_depth(j)
— Normalised circuit depth of job j, computed as estimated_depth / MAX_DEPTH_NORM (capped at 1.0). Deeper circuits incur a scheduling penalty.
w_p
— User-priority weight (default 0.40).
w_f
— Fidelity weight (default 0.30).
w_e
— Error-rate penalty weight (default 0.20).
w_d
— Circuit-depth penalty weight (default 0.10).

1.3   Allocation Criteria

  • Qubit capacity: the QPU must have qubit_count ≥ job.min_qubits.
  • Availability: the QPU must be in ONLINE status with no currently assigned circuit.
  • Score maximisation: among all eligible QPUs, the one with the highest composite score is selected.

1.4   Dynamic Reallocation

⚠ WARNING — QPU Status Transition Before initiating dynamic reallocation, verify that displaced circuits have not produced partial results. Partial measurement data from interrupted circuits is non-recoverable and must be discarded.

When the Resource Monitor detects that a QPU has transitioned to DEGRADED status (e.g., due to increased error rates or reduced coherence times), the following reallocation procedure is executed:

  1. The reallocate_degraded() method identifies all circuits currently assigned to degraded QPUs.
  2. Affected circuits are unassigned from the degraded QPU and their status is reset to QUEUED.
  3. The circuit is re-inserted into the priority queue for rescheduling on a healthy QPU.
  4. The schedule_pending() method is invoked to attempt immediate reallocation.

2   Circuit Queue Management Subsystem

2.1   Queue Architecture

The circuit queue is implemented as a min-heap priority queue keyed on (priority, submission_timestamp). This guarantees that higher-priority circuits are dequeued first, with FIFO ordering among circuits of equal priority.

2.2   Maximum Queue Depth

The maximum queue depth is 512 circuits (constant MAX_QUEUE_DEPTH = 512). This limit is enforced at submission time.

2.3   Priority Levels

LevelEnum ValueTypical Use Case
CRITICAL0Safety-critical in-flight computations, real-time control loops
HIGH1Hybrid orchestrator iterations, time-sensitive optimisations
NORMAL2Standard user-submitted circuits
LOW3Batch processing, non-urgent analyses
BACKGROUND4Calibration circuits, diagnostic routines

2.4   Circuit Cancellation Strategy

⚠ CAUTION — Cancellation of Executing Circuits Cancelling a circuit in EXECUTING status sets its state to CANCELLED, but the QPU may continue processing until the executor polls the status flag. Do not assume instantaneous termination. Verify circuit status after cancellation before resubmitting dependent circuits.

Cancellation behaviour depends on the current circuit status:

Current StatusCancellation ActionQPU Released?
QUEUEDImmediately removed from queue; status → CANCELLEDN/A
SCHEDULEDStatus → CANCELLED; assigned QPU releasedYes
EXECUTINGStatus → CANCELLED; executor expected to abort on next pollOn executor acknowledgement
COMPLETEDNo action (already finished)Already released
CANCELLEDNo action (idempotent)Already released

2.5   Queue Overflow and Eviction Policy

When the queue is at maximum capacity (512 circuits) and a new submission arrives:

  1. The system identifies the lowest-priority queued circuit in the heap.
  2. If the incoming circuit has a strictly higher priority than the worst queued circuit, the worst circuit is evicted (status set to CANCELLED) and the new circuit is enqueued.
  3. If the incoming circuit does not outrank any existing entry, a RuntimeError is raised and the submission is rejected.

2.6   Re-submission and Failure Handling

When a circuit fails during execution:

  1. The job status is set to FAILED and the QPU is released.
  2. The resubmit(circuit_id) method checks whether retries remain (retries < max_retries, default 3).
  3. If retries remain, the circuit is re-inserted into the queue with incremented retry count and status reset to QUEUED.
  4. If retries are exhausted, the circuit remains in FAILED status and the owner is notified via the monitoring alert subsystem.

2.7   Circuit Lifecycle

Figure 3 shows the full state machine governing circuit lifecycle. All three active states (QUEUED, SCHEDULED, EXECUTING) may transition to CANCELLED. The FAILED state supports a retry loop back to QUEUED, limited by max_retries.

Alerts are triggered when any metric sample falls outside the configured threshold bounds (set via monitor.set_threshold(metric, low, high)).

4   Quantum-Classical Hybrid Orchestration Subsystem

4.1   Purpose

The Hybrid Orchestrator manages iterative quantum-classical algorithms (e.g., VQE, QAOA) where a classical optimiser and a parameterised quantum circuit exchange data in a closed loop until convergence.

4.2   Orchestration Protocol

The iterative loop depicted in Figure 5 proceeds as follows:

  1. Initialisation: The classical optimiser provides an initial parameter vector θ₀ to the quantum circuit template.
  2. Quantum execution: The parameterised circuit is submitted to the Circuit Queue with HIGH priority. The QPU Allocator selects the best available QPU and the circuit is executed.
  3. Measurement return: Raw measurement results are post-processed through error mitigation (zero-noise extrapolation, measurement-error mitigation matrices) before being returned to the classical stage.
  4. Classical update: The optimiser (e.g., COBYLA, L-BFGS-B, SPSA) evaluates the cost function, computes new parameters θ_{k+1}, and checks convergence: |Δθ| < ε.
  5. Loop or terminate: If converged or max_iterations reached, the job completes and the final result is stored. Otherwise, the loop repeats from step 2.

4.3   State Synchronisation

Parameter vectors are serialised as tuples alongside the circuit template at each iteration. The HybridJob dataclass maintains the canonical state:

  • parameters — current parameter vector (updated after each classical step)
  • current_iteration — iteration counter
  • converged — boolean convergence flag
  • result — final measurement result (populated on convergence)

4.4   Error Mitigation Strategies

StrategyDescriptionWhen Applied
Zero-Noise Extrapolation (ZNE) Execute circuit at multiple noise scale factors and extrapolate to the zero-noise limit Post-measurement, before feeding back to classical optimiser
Measurement Error Mitigation Apply an inverse confusion matrix derived from calibration circuits to correct readout errors Immediately after QPU readout
Dynamical Decoupling Insert identity-equivalent pulse sequences during idle qubit periods to suppress decoherence Circuit compilation stage (pre-execution)
Symmetry Verification Post-select measurement results that satisfy known symmetry constraints of the target Hamiltonian Post-measurement filtering

4.5   Convergence Criteria

A hybrid job terminates when any of the following conditions is met:

  • The classical optimiser returns None (signalling convergence to the orchestrator).
  • The parameter update norm falls below convergence_threshold (default 1×10⁻⁶).
  • The iteration count reaches max_iterations (default 100).

5   API Reference Summary

5.1   Circuit Submission API

circuit_id = mgr.queue.submit(
    circuit_data=my_circuit,
    priority=CircuitPriority.NORMAL,
    owner="user@example.com",
    estimated_depth=250,
    min_qubits=8,
    max_retries=3,
)

5.2   Status Retrieval API

status = mgr.queue.get_status(circuit_id)    # → CircuitStatus enum
job    = mgr.queue.get_job(circuit_id)        # → CircuitJob dataclass (full details)

5.3   Cancellation API

success = mgr.queue.cancel(circuit_id)        # → bool

5.4   QPU Management API

mgr.allocator.register_qpu(QPUDescriptor(
    qpu_id="QPU-0",
    qubit_count=127,
    avg_gate_fidelity=0.9985,
    t1_us=120.0,
    t2_us=95.0,
    error_rate=0.005,
))
mgr.allocator.update_qpu_status("QPU-0", QPUStatus.CALIBRATING)
qpus = mgr.allocator.list_qpus()             # → List[QPUDescriptor]

5.5   Monitoring API

mgr.monitor.set_threshold("gate_fidelity", low=0.990, high=None)
mgr.monitor.set_threshold("error_rate", low=None, high=0.020)

mgr.monitor.record(MetricSample(
    timestamp=time.time(),
    metric_name="gate_fidelity",
    value=0.9982,
    qpu_id="QPU-0",
))

latest = mgr.monitor.get_latest("gate_fidelity", qpu_id="QPU-0")
alerts = mgr.monitor.get_alerts(acknowledged=False)

5.6   Hybrid Orchestration API

job_id = mgr.orchestrator.create_hybrid_job(
    classical_fn=my_optimizer,
    circuit_template=ansatz,
    initial_params=[0.1, 0.2, 0.3],
    max_iterations=50,
    convergence_threshold=1e-5,
)
done = mgr.orchestrator.step(job_id)
done = mgr.orchestrator.feed_result(job_id, measurement_result)
hjob = mgr.orchestrator.get_job(job_id)
